# 一年后Claude不需要Harness工程了？产品和工程负责人爆料：搭建Agent的最终难关是基础设施壁垒；Harness和模型正高度配对

“ 一年后，我们希望实现极致的简化…… 在那个世界里，你不需要再考虑‘Harness 工程’，不需要纠结如何构建工具。你会删掉很多脚手架代码，也不需要那么多提示词工程。”

“我认为基础设施才是大多数人最终撞上的壁垒，尽管他们原以为 Harness 工程和挖掘模型潜力才是最难的。”

“大家都在试图压榨每个模型的每一丝性能，针对每一点细微差别进行‘ Harness 工程。’”

今天 Claude 平台的产品负责人 Angela Jiang 和工程负责人 Katelyn Lesse 的访谈播客上线了！

这周三，Anthropic 刚在 Code with Claude 开发者大会上发布了 Managed Agents 的梦境（Dreaming）、多智能体编排和结果验收（Outcomes）等全新功能。 [Claude的Agent四件套终于来了！Agents是新一代应用形式，Harvey任务完成率暴涨6倍！网友：有一批创业公司要倒下了！](https://mp.weixin.qq.com/s?__biz=MjM5ODI5Njc2MA==&mid=2655939196&idx=1&sn=7448d074aaca25a289a85decc48d62a7&scene=21#wechat_redirect)

这场播客正好就讨论了为什么 Anthropic 要开发 Managed Agents 和它是如何被构建和使用的。

当然 Claude 平台的发展轨迹、对 OpenClaw 的看法、搭建 Agent 的难点、团队使用 Agent 的形式、以及一年后 Claude 将是什么样子，两位嘉宾和主持人都有聊到。

对于搭建 Agent，两位嘉宾讲到“ 人们普遍觉得 Harness 工程是最难的部分”，但其实“基础设施之墙”才是他们最终会撞上的难关。她们接触过很多人，Agent 原型一旦投入生产并进行规模化，就会面临沙箱断连、内存丢失、异步处理等问题。

而 A nthropic 推出 Managed Agents 的原因就是不想再重复踩基础设施的坑了，所以把文件系统、Skills 和记忆都集成到了云端。

她们还指出一个趋势，Harness 和模型正变得高度配对。几个月前，还在流行构建一个通用 Harness，然后随时切换模型。而现在，各家实验室都在采取不同的技术和视角，用 Harness 工程来竭尽所能地提升 模型的每一丝性能。

Managed Agents 的新功能多智能体编排就被人们用来 尝试构建不同的 Harness 技术了，比如将“执行”与“建议”分开的智能体，有内容生成和内容审核的“对抗” 智能体 和适合于找 Bug 的“蜂群（swarm）”智能体。

对于 Claude 的未来，在主持人提问 她们如何看待 OpenClaw 这种智能体形态时，她们非常肯定 Claude 也会向容易部署的 Agent 方向进化，不过起步时重点放在了基础设施上，Anthropic 希望用更安全的方式封装“智能体身份”类基本组件。

而当主持人让她们推理想象一下一年后的 Claude 会是什么样子的时候，她们的第一反应是“到那时候我们可能就不坐在这儿了，得求着它办事。”

她们希望一年后 Claude 平台能实现极致的简化，“Claude 会变得非常了解自己，能自动判断该用哪个模型，如何启动子智能体。”用户关心的参数只有结果和预算，Claude 可以动态编写自己，在两个参数下找到最优解。

在那个世界里，现在的很多创新都会消失，智能体是持续运行、不断自我重构的。

访谈中还聊了很多，全部内容就在下边，enjoy~

Claude 发展主线：越来越自主，让用户获得最好的结果

主持人： Angela, Katelyn ，欢迎来到节目。

Angela/ Katelyn ：谢谢邀请。是的，谢谢。

主持人：介绍一下，你们两位都在 Anthropic 负责平台工作。Angela 是 Claude 平台的产品负责人， Katelyn 是工程负责人。我很期待和你们交流，因为你们最近发布了很多东西，比如刚推出的 Claude Managed Agents，还一直在上线新功能。我认为这正处于一个非常有趣的时刻，让我思考对于一家模型公司来说，AI 领域的“平台”到底意味着什么。在 GPT-3 时代，平台只是个补全端点，发个提示词拿个回复。后来变成了带工具调用和对话会话的端点。而现在，通过 Managed Agent，你本质上是在一台拥有内存等各种功能的电脑上运行 Claude。我想请你们帮我梳理一下这个发展轨迹，以及在 AI 领域构建平台的深层含义。

Angela：没错，我觉得你的描述非常准确。随着大语言模型（LLM）技术的演进，最初把它放在 API 后面很有趣，大家当时觉得“哇，我能做这么多事”，虽然现在回头看可能会觉得很基础。后来我们越来越倾向于“有状态”的世界，因为你需要持久化会话状态来确保模型性能不断提升。这其实是一条主线：随着我们改进 Claude，它变得越来越自主，我们需要将平台演进为更高阶的抽象，目标是帮你通过模型获得最好的结果。

刚开始大家都在探索，谁也不知道人们会用 LLM 做出什么，所以要提供尽可能多的可能性。随着使用场景逐渐明确，人们开始构建产品、构建智能体，越来越多的客户跑来问：“我该如何发挥 Claude 的最佳性能？”、“该如何设置工具？”、“如何运行循环？”等等。还有一大群新用户，他们希望这些功能可以“开箱即用”。为了确保 Claude 能产生最佳效果，我们不断丰富平台，包括状态管理、新增的工具，以及各种云端组件。核心使命始终没变：让一切变得尽可能简单。我认为未来平台的哲学，最终会归结为：提供一套基础设施和基本组件 （ primitives） ，让你以最少的工作量、最快的速度拿到结果。

不再重复踩基础设施的坑了！Managed Agents 诞生的原因

主持人：那么，今天的 基本组件 是什么？或者说，Claude Managed Agents 中包含哪些 基本组件 ？

Katelyn ：Managed Agents 是基于我们现有的 基本组件 构建的，比如 Messages API。在这个 API 内部，我们做了很多创新。如果你愿意，可以直接获取 token 的输入和输出，但你也可以使用我们内置的工具，比如代码执行（开启沙箱运行任务）、网页搜索等。我们将这些强大的功能整合进一套基础设施中，以实现我们认为的 Claude 最佳产出。

主持人：听你们说这些，我感受到一种 “时间通缩”感，也就是说，我的时间在未来会变得越来越值钱。因为我们自己内部也在构建智能体产品，有些是自用的，有些是给客户的。为了实现这些，我们用了几台 Mac Mini，让 Claude 在上面循环运行，写了上千行 Python 代码。这在很大程度上镜像了你们正在做的Managed Agents 。所以我和很多开发者都有这种感觉：也许我们应该直接等着你们做出来。我不确定界限在哪，如果我想构建一个智能体，最符合你们发展方向的最佳路径是什么？

Angela：这正是平台业务的相似之处。像你这样的客户会纠结：是现在就动手做以满足紧迫需求，还是等平台免费提供，避免重复造轮子。而且，搞基础设施太痛苦了。部署服务器这种事真的很糟，真不敢相信你们竟然一直在亲手干这个。（笑）

Katelyn ：我补充一点，我们之所以构建Managed Agents，是因为 Anthropic 自己也经历过无数次迭代。我们构建过能自主运行的智能体产品，踩过了无数次搞定基础设施的坑，最后我们觉得：够了，别再为自己重复造轮子了。我们要结合所有经验，一劳永逸地做出一套真正好用的东西。对于在几台 Mac Mini 上跑任务的人来说，这可能暂时够用，但如果你要大规模运行智能体，搞定基础设施就会变得极具挑战。

Angela：关于产品哲学，我们有两点想法。第一，Managed Agents 的设计是模块化的。我们对某些部分有“定见”（opinionated），比如我们希望它与 Claude 模型深度结合。例如，我们非常希望 Claude 以特定方式使用文件系统。我们还非常看重 Skills ，并希望我们的框架对此有明确的设计导向。所以我们很看重文件系统、技能这些基本组件，尽管它们看起来很基础。与此同时，我们也看到人们在尝试其他方法，我们希望帮大家从一开始就走在正确的轨道上。虽然现在可能还没开放所有功能，但在设计上，我们努力保持灵活性，让人们能添加不同的组件，因为我们意识到这套 API 不可能解决所有原始构建中的问题。第二点是，我们在设计这些东西时会发布博客和 实现 参考。如果你想参考我们的结构但在 Messages API 上自行构建，那是完全没问题的。

Harness 与模型正变得高度配对，只为榨干模型每一分性能

主持人：这正是我刚才想说的。我们团队目前就在纠结：对于为客户提供智能体这件事，如果使用 Claude Managed Agents，会不会被锁死？现在我们就像有个游乐场，有服务器，有 Mac Mini，可以直接把东西传给 Claude。它能做任何 Claude 代码能做的事，有文件系统，有浏览器。如果我们想换成 GPT-5.5 或 Gemini，非常容易。大家担心会失去灵活性，没法做想做的所有尝试。还有一个担忧是，有些新功能可能会先出现在 Claude Code本身，而Managed Agents 要等一阵子才能跟上，这会让我们无法处于技术前沿。你怎么看？

Katelyn ：我们内部工作方式的一个优点是：我们负责平台，而 Anthropic 所有的第一方产品，如 Claude Code， 都是直接构建在和大家一样的平台上。所以我们花了大量时间与内部团队合作，共享创意并启用新功能。随着时间推移，你会看到Managed Agent 和 Claude 代码之间的差异越来越小，因为它们都基于同一套基础设施。

Angela：关于你提到的“模型绑定”担忧，这非常合理。但我认为我们正处于一个演进阶段。几个月前，构建一个通用的 Harness 然后随时切换模型是标准做法。对于旧一代模型，这还算合理。但对于下一代模型，你会看到各家实验室都在采取不同的技术和视角。为了最终给客户交付成果，那种“极致通用”的抽象层开始失效了， Harness 和模型正变得高度配对。你当然还需要冗余，也可能在某些地方用其他模型，但这种替换更倾向于发生在“智能体”层级，像 Harness 加 模型的组合，而不是在最底层的通用 Harness 里随便拔插模型。

主持人：这很有趣。像 Cursor 这样的公司也是这么做的吗？他们是为每个模型准备独立的 Harness ，还是用通用 Harness 来回切换？

Angela：我不太确定 Cursor。但我的直觉是，我们交流过的一些团队也有类似看法。这主要是因为大家都在试图压榨每个模型的每一丝性能，针对每一点细微差别进行“ Harness 工程”。举个内部的例子：我们最近在 Managed Agents 中上线了“记忆”功能。我们自己尝试了多种不同的 Harness 方案，结果发现每种方案的评估（eval）表现差异极大。这说明只要通过“ Harness 工程”把对的组件拼在一起，就能获得巨大的性能提升。如果把这个思路推广到所有模型、实验室和供应商，这里面蕴含着巨大的超额收益（alpha）。所以，如果其他团队也在尝试这种“单元化绑定”，我一点也不意外。

主持人：这种路径依赖很有趣。你在请求响应、工具调用或是否让模型使用文件系统上做出的微小选择，最终会改变所有这些不同模型的轨迹。

Angela：没错。当时看来可能只是个微不足道的注脚，最后却影响巨大。

选对路径和组件，决定模型未来走向

主持人：你认为这最终会影响模型的泛化能力吗？也就是在某种程度上，它们会因为各有所长而被困在特定的 “赛道”里。比如 Claude 擅长文件系统，GPT 擅长别的。这种绑定会如何影响模型的性格和行为？

Angela/ Katelyn ：我认为这确实倾向于让模型产生某种锁定。因此，我们最终视为 “正确路径”和“正确基本组件”的东西，都需要经过非常细致的考量。在某些模型时代，模型变得非常擅长推理，甚至到了在推理层面过度优化的地步。还有另一种视角，即我们也希望它像计算机一样好用，也许“计算机”这部分才是真正有趣的地方。因此，深入思考这些基本 组件， 无论我们想对还是想错，至少这一思考过程会引导我们走向其中一条路径。我觉得很难说哪条路最终才是正确的，但我确实认为这其中存在很强的路径依赖。所以，对于选择包含什么，或者给模型提供什么样的原生能力，必须深思熟虑，这至关重要。

主持人：有没有哪些路径依赖是你们不得不去撤销或推倒重来的？

Angela/ Katelyn ：呃，大概有吧。关于 Anthropic 层面我不能透露太多，我也才来几个月，但我可以想象肯定发生过这种情况。我的意思是，即使在其他实验室，我们也做过实验，我们需要关注的基础组件一直在变化。你有时会陷入局部最优解，然后重新思考：也许存在一种更通用的方法。

谁需要用 Managed Agents？ 只有两类核心受众

主持人：是的，很有趣。我想先退后一步，问一个也许一开始就该问的问题： Claude Managed Agents 到底是给谁用的？就像我今天早些时候设置了一个，我们 Every 内部已经有人在生产环境使用了，我自己今天也弄了一个。我非常喜欢你们提供的那个快速入门聊天体验以及一些示例，给我的感觉是，即便我不懂技术，也可能想用它来搭建一个智能体。虽然可能有点复杂，但我实际上，抱歉这么说，我是在 Codex 的应用内浏览器里完成的。我让 Codex 驱动Managed Agents 的设置，很快就弄好了一个 Slack 机器人，这真的很酷。所以，当你们设计 Managed Agents 时，预想的受众是谁？

Angela/ Katelyn ：很有意思，你说得对，尤其是那个快速入门体验，我们当时非常坚持要上线。倒不是为了让非技术人员去构建智能体，而是为了让任何人（无论懂不懂技术）都能理解这些基本组件。比如它能做什么，以及它是如何组合在一起的，这更像是一种教育引导。但关于受众，我们考虑的是几个不同的方面。一方面，我们看到公司内部人员在构建自动化工具，或者非常强大的平台和系统。比如有人说想要一个完整的端到端软件开发平台，Managed Agents 就是这类需求的完美解决方案。或者想自动化一个小流程，比如让法务审核营销文案，你没必要为此重新实现内存之类的功能，这样你可以快速起步。另一个核心用户群是那些将 AI 构建进产品并提供给最终客户的人。在这种情况下，你确实需要大量定制，想要为产品打造强大的功能，但我们坚信，不应该把工程资源浪费在基础设施和各种繁琐的 Harness 工程微调上。

主持人：为什么我们不一个月前聊呢？那样能帮我们省下太多时间了。（笑）

Angela/ Katelyn ：看来我们以后确实需要多聊聊。（笑）

原型易做，落地难！智能体的终极阻碍：基础设施壁垒

主持人：我有点好奇，基础设施算是一方面，但当你看到人们搭建智能体时，你觉得他们认为的难点是什么？最后真正的难点又是什么？这两者是一回事吗？

Angela/ Katelyn ：好问题。这可能有点 “辛辣”，但我认为人们普遍觉得 Harness 工程是最难的部分。所以事实上，我们之前发布了智能体 SDK，也就是你们在 Mac Mini 上使用的那个。

主持人：我觉得我们其实只是在用 Claude 的命令行工具，比如 claude-p 。

Angela/ Katelyn ： 哇，那也挺好。 好的，但不管怎样，你们这么做是为了省去构建 Harness 的麻烦，对吧？但我确实发现，很多客户在尝试将原型投入生产并进行规模化时，都会撞上 “基础设施壁垒”。每个人都会遇到同样的问题：要么得让服务器一直跑着，要么得用能自动启停的基础设施，还得存储对话记录，需要安全的沙箱环境等等。比如你在沙箱里运行一个会话，如果沙箱断连了，你的智能体就挂了。所以我认为基础设施才是大多数人最终撞上的壁垒，尽管他们原以为 Harness 工程和挖掘模型潜力才是最难的。

Angela/ Katelyn ：我完全同意。我们接触过很多人，他们原型开发得飞快，非常兴奋，觉得 “成了”。然而还有一类人在不断深挖，想要优化框架。但一旦成型，将其产品化简直是一场噩梦。尤其是那些更有趣、需要长时间运行、异步且更具自主性的智能体。每个人都会撞墙，这正是我们构建这套系统的初衷。

OpenClaw 是 Claude 进化的未来方向

主持人：我觉得 OpenClaw 算是智能体的一种形态。特别是它带给我们的体验：在 Slack 里有一个始终在线的智能体，它有自己的性格，负责特定的工作领域。你们觉得这会是未来的方向吗？一个可以一键部署在 Slack 里的智能体，我可以配置其内部逻辑，但完全不需要考虑技术基础设施。因为我觉得你们已经有了雏形，但从现在的 Managed Agents 到 Slack 里那种始终在线的东西，还有很长的路。这属于平台的职责范围吗？还是说这太偏向产品端了？

Angela/ Katelyn ：不，这绝对是我们想做的。我们起步时重点放在基础设施上，因为那是痛点所在。但是，是的，在其进阶形态下，我们确实希望让部署这些智能体变得极其容易。我们已经迈出了一些初步的小步子。比如我们将 “保险库（Vaults）”作为基本组件之一，用来存储密钥、OAuth 令牌和凭据。这是一个起点，解决了一些底层问题。但一旦你能以更安全的方式封装这些“智能体身份”类的基本组件，并且能轻松处理、与整个系统协作，我认为很自然地，我们会进化到这一步：你要么一键集成 Slack，要么直接告诉 Claude “添加 Slack”，它就会处理好一切，然后你的小机器人就在 Slack 上联系你了。

主持人：我太喜欢了，迫不及待想看到那个世界。（笑）目前智能体最好的内部使用场景是什么？因为现在大家都在讨论：大家都在用 Codex 或 Claude Code，但现在有了云端智能体，公司里的每个人都可以拥有自己的智能体，还有团队级和公司级的。当人们制作非常有用的内部智能体时，你观察到了哪些模式？它们的功能和形态是怎样的？

Angela/ Katelyn ：我会说，我们看到了一些领先的 AI 驱动的公司示例，比如 Stripe 构建了 Minions，并将其视为工程师可以使用的端到端开发平台。我觉得 Ramp 也做了类似的事情，我们也一样。我们构建了一些内部平台，让智能体在后台运行，我可以在 Slack 或任何地方与它们交流。在某种程度上，这其实只是在托管智能体之上加了很薄的一层，不需要做太多就能实现。

主持人：我也这么想，我看 Minions 或 Ramp 的做法时就在想：这真的有用吗？在公司里搞一个这种人人都能用的简易编程智能体，为什么不直接在 Slack 里装个 Claude 应用呢？

Angela/ Katelyn ：我觉得这类平台与我们内部做法的区别在于，你可能需要对开发环境进行大量定制，让智能体在其中运行并能够验证其更改。比如： “这就是我们的 CI/CD 流程。”

主持人：没错。

Angela/ Katelyn ：所以，对于很多人来说， Claude Code 确实是个极好的工具，配合智能体使用非常棒。但如果你想做更完整的端到端开发，并且想加入更多自定义的东西，那么可以从 Managed Agents 开始，在上面构建一层。

提升团队生产力的 Agent 形态，还在探索中

主持人：在我看来，有些特定的团队协作场景，让 Managed Agent 这种形态变得比单纯在 Claude Code 里运行更重要。也许技术上你可以同步每个人的技能配置，但大家共用一个专门负责某项任务的智能体，这种模式似乎更奏效。

Angela/ Katelyn ：很高兴你提到这点，因为这也是我们看到巨大机会的领域。如你所言，无论是开发者还是非开发者，现在都有很多提升个人生产力的工具。你可以用很多工具来让自己更自动化、更具杠杆效应。但一旦涉及到团队层面，一切就会变得极其复杂。

首先，它显然不能只跑在你的笔记本电脑上。即便你把它放上云端，那通常也只是为了你自己方便。但如果是我们三个人想要几个能互相协作、完成端到端流程自动化的智能体呢？尤其是对于那些你认为会被 AI 彻底重塑的复杂流程，你确实需要这种团队导向。这需要发生在一个比单一智能体更高一点的抽象层级。一些团队在探索的多智能体架构非常令人兴奋，但这需要建立在一个大家都能控制的平台之上。Vercel 的 G 对此有很好的见解，他将公司内部描述为“AI 软件工厂”。我觉得这就是正确的心态。这会造就一个极高杠杆的组织，不仅提升个人效率，更提升公司每一个流程的生产力。

Agent 帮 Anthropic 法务审核营销文案

主持人：我还是想回到智能体使用场景上。既然有了人人可用的编程智能体，你还看到哪些非常有用的场景？

Angela/ Katelyn ：我们见过不少。有趣的是，我们可以与内部不同职能的团队合作，帮他们寻找场景，我们也因此学到了很多。比如我之前提到的那个 “法务审核营销文案”的例子，这非常真实。

主持人：是的，非常真实。哪怕是很基础的智能体，只要配置得当，就能让人大开眼界。所以你见过那样的例子。

主持人：它具体是怎么运作的？是有一个法务智能体盯着营销的一举一动，然后随时喊停吗？

Angela/ Katelyn ：更像是这样：作为一个营销人员，我写了一段文案。以前你可能得开个工单，求法务审一下。但现在，你把它提交给这个基于智能体的小应用。智能体会先进行初审，然后放进法务的收件箱，并注明 “已完成第一轮预审”。智能体甚至可能直接给出反馈：“营销部，这篇没问题。” 或者它会提示：“不，这篇还需要人工额外审核。” 这也是一种“轻量化”的应用。你我有权限，都能看到产出并协作。

主持人：等等，为什么这不算是一个 Skill ？

Angela/ Katelyn ：它完全可以是一个 Skill 。你构建一个 “法务审核智能体”，给它配置 MCP 服务来获取外部上下文，给它配置理解规章制度的 Skill 。你把这些整合在一起，然后启动一个会话。但你还需要最后一块拼图，也就是我说的那个“轻薄层”，即上层的交互形态，让不同的人能协同工作，让多个智能体能参与到系统中。所以我认为这比 Skill 的范畴更广一点，因为你仍然需要合适的交互形态。

Angela/ Katelyn ：另一个核心原因在于，你确实需要 “人工介入（human in the loop）”。如果是全自动化，纯靠 Skill 自审，那当然可以。但如果你需要人工审核和确认，尤其是涉及到法律事务，通常需要身份验证。为了自动化整个流程，你需要智能体去执行任务。由于需要启动独立的会话来处理，这种“缝合”工作是无法通过单一 Skill 实现的。

Anthropic 内部：自助修改 Agent

主持人：很有趣。我想再深入一点。所以对你们来说最佳实践是：创建一个智能体，任务是确保营销文案能快速通过法务审批。有时它能秒批，有时转给法务。理想情况下，它会不断进化，能处理的事也越来越多，对吗？一旦智能体构建完成，关于谁拥有它的最佳实践是什么？因为我们发现，如果没有专人负责，智能体很快就会过时，最后变成一个在那儿瞎忙活但效果不佳的 “僵尸”。此外，即使它勉强能用，法务也常会说：“你让我审这个，但这其实没必要审，咱们改改提示词吧。”那么在理想状态下，这一切是如何运作的？

Katelyn /Angela：这其实很有意思，关于应用形态，也就是我们最初构建的那层应用，我们的一个团队专门负责这个，他们会深入用户团队了解需求。原本以为交付后就可以不管了，结果发生了一件很酷的事：使用这些工具的团队成员会说：“噢，我希望这个小细节能调整一下，或者那个地方能改进。”然后他们直接打开 Claude Code 动手修改了代码。

主持人：那你们团队负责审核 PR（拉取请求）吗？还是直接就合并了？

Katelyn /Angela：如果是我们实际拥有的系统，通常由我的团队负责审核。但没错，人们可以自助修改，这非常酷。我认为很多团队和公司目前仍处于这个阶段。回到刚才说的，比如 Stripe 有 Minions 项目，他们有一个庞大的开发者效能团队。我们以前在 Stripe 工作过，所以了解他们。他们显然投入了巨大精力来构建这类平台和工具。所以我觉得，虽然 Managed Agents 或在平台上二次开发非常强大，但你仍然需要企业内部那些有“AI 觉悟”的技术人员，去构建出真正契合业务目标的优秀成果。

对话改 Agent，调用“套娃式” Managed Agents

主持人：挺有意思。我很喜欢 “任何人都能开 PR”这个点，因为大家都在用 Claude Code。我在和负责基础设施的人交流时发现，你知道那个背后插满匕首还要保护别人的表情包吗？（笑）基础设施人员现在就处于这种状态，因为任何人都能提交 PR。你怎么处理这种情况？如何把它做好？因为理想情况下，你肯定希望法务能提交 PR 改进智能体，但有时他们可能会提交一些浪费时间的蠢东西。那么，从组织或技术层面看，什么是既能实现协作又不会毁掉你们生活的正确方式？

Katelyn /Angela：针对 Katelyn 举的这个例子，我们实际上在 PR 层之上构建了几层抽象。刚开始确实是直接开 PR，但为了防止用户“搬起石头砸自己的脚”，现在通常是：无论营销还是法务团队，他们拥有的智能体，其交互方式更多是直接通过 Claude 进行。他们大部分时间在和 Claude 对话，然后由 Claude 确定修改的核心逻辑，这样他们就不必接触最底层的代码，从而避免引发复杂问题。

主持人：他们是在跟 Claude 聊天，还是在用 Claude Code、Claude Chat 或协作模式？

Katelyn /Angela：那是我们构建的一个专门的 Claude 实例，它本身也是一个托管智能体。所以在这个架构里，真的是“套娃式”的 Managed Agents。我们发现，如果我们对托管智能体的每一个变体进行调优和提示词设定，就能帮用户解决问题的不同部分。最终对于营销或法务人员来说，界面非常简单，他们感觉只是在跟 Claude 说话。但在底层，是多个 Claude 在相互配合，完成那些人类不需要理解的复杂工作。

多智能体编排新玩法：分角色、“对抗”、群体找 Bug

主持人：很有趣。你们刚刚发布了多智能体编排（ Multi-agent Orchestration）。目前人们用它做的最酷的事情是什么？

Katelyn /Angela：其中一个很有趣的用法是构建不同的“Harness 技术”，我个人对此非常兴奋。人们实验了不同的技术，比如我们最近做的“顾问策略（advisor strategy）”，通俗点说就是将“执行”与“建议”分离。还有一种模式是让两个智能体“对抗”：一个生成内容，另一个进行对抗性审核。也可以将任务拆解成很多微小的碎片再重新组合。还有类似于“N 选 1（Best-of-N）”的模式，等等。每种架构或策略都适用于非常具体的场景。比如，有些更适合深度或广度研究；有些类似“蜂群（swarm）”的模式则更适合找 Bug。看到这些基本组件像乐高积木一样被组合成更高阶的架构，并获得更显著的效果，这真的很令人兴奋。这表明你可以在多个抽象层级上不断优化。

主持人：你如何判断一个智能体是否成功？如何衡量它的成效？

Katelyn /Angela：大家都说烂了的评测（Evals）当然算一种。但我们非常推崇的一个方向是“可验证的结果”。在这个问题上我们挺有主见的。如果我们谈论平台的终局，按照这个哲学，应该是：一切都应压缩为“结果”和“预算”，仅此而已。其他所有过程都应该自动为你解决。所以，我们虽然仍有评测，以及一些特定领域的衡量标准（比如代码评测看 PR 是否成功合并），但随着我们不断进步，“结果”实际上会变成一个由人类定义的规范，而我们关心的是模型能否准确解读并不断自我修正。

主持人：比如： “Claude，给我赚 10 亿美金，预算 10 块钱。”

Katelyn /Angela：没错。（笑）还得补一句：“不准出错，开始吧。” 也许 Mythos 能做到。

Anthropic 开发专属 Skill ，让 Agent 自动升级

主持人：我还遇到一个问题：智能体过时得很快。有时是因为没专人管，有时是因为跑的是旧模型或旧架构。感觉智能体需要一个 “生命周期终点”，我们甚至讨论过给它们办个葬礼，在网站上弄个“退役智能体”页面。在大型公司里，如何管理那些还在 Slack 里每周蹦跶但已经过时的智能体？如何确保退役它们的速度跟创建它们一样快？

Katelyn /Angela：我们做的一件事是开发了专门的 Skill ，帮助智能体在新模型发布时进行自动升级。我们投入了大量精力让这个过程变得更简单。那些最狂热的 AI 信徒甚至会让智能体去监控其他智能体，看它们是否过时。但对于客户，我觉得最有趣的场景是：新模型出来了，我需要升级智能体，甚至直接弃用旧的，因为新模型能构建出强大得多的智能体。人们必须理解，这种升级和迁移就像是一种“破坏性变更”，需要投入精力。如果有评测体系，这个过程会容易些。但我们努力做的，就是提供工具让这个过程不再痛苦。然后你可以选择用更多的智能体来实现自动化。

一年后的 Claude：极致简化，动态编写自己

主持人：一年以后，我们再次坐在这里。你觉得平台会变成什么样？我能做什么？到那时会有什么不同？

Katelyn /Angela：你想先说吗？一年在这一行可是很久的。

主持人：我们离 Claude 赚到 10 亿美金还有多远？（笑）这才是我想问的。

Katelyn /Angela：到那时候我们可能就不坐在这儿了，得求着它办事。（笑）我是说，我们想尽可能接近那种状态。一年后，我们希望实现极致的简化，这可能意味着更高阶的抽象。用户关心的参数将只有“结果”和“预算”。Claude 会变得非常了解自己，能自动判断该用哪个模型，如何启动子智能体。在那个世界里，你不需要再考虑“Harness 工程”，不需要纠结如何构建工具。你会删掉很多脚手架代码，也不需要那么多提示词工程。现在的很多创新发生在“Harness架构”这种高层级，这很有趣，但以后很多东西都会消失。你不用选模型，不用想架构，因为 Claude 已经经历了足够的迭代，能够动态地“编写自己”，在“结果”和“预算”这两个参数下找到最优解。虽然一年内不一定能完全实现，但在一定的误差范围内搞定“结果”部分还是很有希望的。

Katelyn /Angela：挺酷的。我也给一个稍微“无聊”点的答案：在那个世界里，如果 Claude 正在动态生成你的智能体，那么平台本身必须具备极强的扩展性。我们的团队会致力于寻找正确的抽象方式来实现这一点。我会确保人们进出 Claude 的“Token 流”顺畅，因为系统扩展到了足以满足这种需求。在那个世界里，智能体是持续运行、不断自我重构的。你需要一个能处理长时请求、处理各种形态任务的系统。一年后我们再聊时，我希望平台自身的扩展能力永远不会成为人们创新的阻碍。

主持人：太棒了，我很期待。非常感谢两位参与，我学到了很多。

Katelyn /Angela：谢谢邀请。
